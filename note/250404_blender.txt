

# 렌더링 프로퍼티


## 렌더링 엔진

eevee 가 속도는 빠르지만 그림자나 빛처리가 아쉽다
작업할때 속도를 위해 사용

cycles
프린트, 영상물 최종처리 할때는 싸이클이 훨씬 퀄리티가 높음.



## 디바이스를 cpu, gpu 로 고르는 차이는 (이해하기 쉽게 비유하자면)

cpu : 연산작업은 엄청높지만 병렬수가 적음
gpu : 연산량 자체는 낮지만 병렬처리 숫자가 엄청나게 많음

인공지능에 gpu가 중요한 이유가 병렬처리가 엄청 중요함



## 샘플링
Max Samples
이미지를 몇개로 나눠서 작업할거냐.
많을수록 오래걸림.
보통 4096은 너무 과하고
해봤자 2048


## Denoise
보통 OptiX



## Light Paths

- Max Bounces

아까 말했다시피
싸이클로 렌더링하면 물체와 물체사이의 빛을 반사해줌 (EEVEE는 그게 안됨)
근데 이 빛 반사가 튕기는 횟수가 있음.

반사를 몇번 할것인지 그게 맥스바운스 (현재 기본값은 12)


- Caustics
비유하자면 수영장 사진을 보면 물결치는 빛 반사 (투명한 물질의 빛 반사)


- Fast GI Approximation
전체 환경광을 계산 (배경의 빛 반사) (이번엔 꺼져있음)



## Simplify 
성능 낮추고 속도 올리는 옵션



## Motion Blur
알잖아 잔상



## Flim
- Transparent
  - Transparent Glass
  유리의 빛 굴절, 통과, 반사


## Performance
- 디바이스 : GPU 로 고정하자
- Threads : 그 쓰레드
- Memory : 알잖아

- Final Render
  - Persistent Data
렌더링 시 이전 프레임의 데이터를 캐시에 보관하여 
다음 프레임에서도 사용할 수 있도록 해주는 기능
주로 Cycles 렌더 엔진을 사용할 때 등장하며, 
렌더 속도를 높이거나 불필요한 재계산을 줄이기 위해 사용됩니다.
예를 들어 애니메이션을 렌더링할 때, 씬의 구조나 쉐이더가 자주 바뀌지 않는 경우, 
매 프레임마다 전체를 재구성할 필요가 없죠.

# Output 프로퍼티

## Format
- resolution
보통 해상도는 2진수에 맞게 적용하는게 처리속도가 빠르다

예를들어 게임 아이콘 등을 만들때도 256, 512 픽셀 등을 맞추는게 좋음.
그래야 쓸모없는 메모리를 차지하지 않음.

300x200 이런식으로 그림 만들면
툴 간의 이미지 공유시 해상도 깨지는경우있음
BMP 비트맵 이미지 (Bit map image) (비트 형식으로 이미지를 저장)
512x256 으로 바뀌면서 짜부러짐

유니티에서 UI 만들때 
아틀라스 (패킹)
하나의 캔버스안에 쑤셔넣어서 씀
아틀라스를 활용하는게 좋음


- Aspect X Y
픽셀 사이즈 비율
픽셀의 크기 조절 (한 픽셀의 비율이 정사각형이 아니라 x:3 y:1 이렇게도 할수 있다는것)

이건 보통 만질일 없고 개념만 알고있으면됨
DV 인 경우 픽셀이 0.9 : 0.9 
1. 캠코더에서 보이는 화면
2. 브라운관 TV
3. 영화관 필름 (옛날) 


- Frame Rate 
초당 프레임수

옛날 영화 필름 24프레임
TV 29.97 프레임
HDTV 30프레임
유튜브 30프레임
일본애니메이션 8~12프레임
